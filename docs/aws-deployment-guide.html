1. Overview

The Playwright Reporter system has four parts:

ComponentRepoRole
Backend APIplaywright-reporter-backendReceives test results, stores metadata in Postgres, files in S3
Dashboard UIplaywright-reporter-uiReact app to view runs, manage projects, get API keys
Reporter packagerocketium-playwright-reporterPlaywright reporter that sends data to the backend
Test suiteautomation-tests-2.0E2E tests — configured to use the reporter in CI
Important: PostgreSQL and S3 work together, not as alternatives. RDS stores run metadata; S3 stores binary artifacts (screenshots, videos, traces).

4. Step 1 — RDS PostgreSQL

4.1 Create the database

  1. AWS Console → RDS → Create database
  2. Engine: PostgreSQL 16
  3. Template: Free tier (or Dev/Test)
  4. DB instance identifier: playwright-reporter-db
  5. Master username: postgres
  6. Master password: choose a strong password (save it)
  7. DB name: rocketium_e2e_runs
  8. Instance class: db.t4g.micro (or larger for production)
  9. Storage: 20 GB gp3
  10. VPC: same VPC you will use for EC2
  11. Public access: No
  12. VPC security group: create new → e.g. reporter-rds-sg
  13. Create database

4.2 Security group (RDS)

TypePortSource
PostgreSQL5432EC2 security group only

Do not open port 5432 to 0.0.0.0/0.

4.3 Note your connection string

postgresql://postgres:YOUR_PASSWORD@<rds-endpoint>.region.rds.amazonaws.com:5432/rocketium_e2e_runs

URL-encode special characters in the password: @ → %40, # → %23, % → %25

postgresql://postgres:MyP%40ss%23word@playwright-reporter-db.xxxx.ap-south-1.rds.amazonaws.com:5432/rocketium_e2e_runs

5. Step 2 — S3 Bucket (Artifacts)

5.1 Create bucket

  1. AWS Console → S3 → Create bucket
  2. Name: e.g. rocketium-playwright-artifacts
  3. Region: same as EC2 (e.g. ap-south-1)
  4. Block all public access: ON
  5. Create bucket

5.2 IAM policy for EC2

Create an IAM role for the EC2 instance:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:GetObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::rocketium-playwright-artifacts/artifacts/*"
    },
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::rocketium-playwright-artifacts",
      "Condition": {
        "StringLike": { "s3:prefix": ["artifacts/*"] }
      }
    }
  ]
}

Attach this role to EC2 so you don't need AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY in .env.

6. Step 3 — EC2 Backend

6.1 Launch EC2

  1. AMI: Ubuntu 22.04 LTS
  2. Instance type: t3.small minimum
  3. VPC / subnet: same as RDS
  4. Security group (reporter-api-sg):
    • SSH 22 — your IP
    • HTTP 80 — 0.0.0.0/0 (or ALB SG only)
    • HTTPS 443 — 0.0.0.0/0 (or ALB SG only)
    • TCP 3000 — your IP only (dev; remove after Nginx)
  5. Attach IAM role from Step 5.2
  6. Launch; note public IP or assign Elastic IP

6.2 Install dependencies on EC2

ssh -i your-key.pem ubuntu@<ec2-public-ip>

curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs
sudo npm install -g pnpm

git clone <your-repo-url>
cd playwright-reporter-backend
pnpm install
pnpm build

6.3 Create backend .env on EC2

cd playwright-reporter-backend

8. Step 5 — Connect Code & Credentials

After AWS infra is up, wire the application code.

8.1 Create project and get API key (Dashboard)

  1. Open https://reports.yourcompany.com
  2. Sign up / Log in
  3. Create project → e.g. "Rocketium Staging"
  4. Copy from project settings:
    • Project ID — e.g. 9gEjLh
    • API Key — e.g. rptr_live_f814f717... (shown once — save immediately)

8.2 Backend — no reporter keys in backend .env

The backend validates per-project API keys from the database (created via UI). Backend needs only: DATABASE_URL, S3 vars, JWT_SECRET.

8.3 Local testing — automation-tests-2.0/.env

REPORTER_API_URL=https://reports-api.yourcompany.com
REPORTER_API_KEY=rptr_live_your_key_here
REPORTER_PROJECT_ID=9gEjLh
REPORTER_TAGS=staging
REPORTER_DEBUG=true

Read by automation-tests-2.0/reporter.config.ts:

apiUrl: env("REPORTER_API_URL") ?? DEFAULT_REPORTER_API_URL,
apiKey: env("REPORTER_API_KEY"),
projectId: env("REPORTER_PROJECT_ID") || "rocketium-staging",

8.4 Playwright config

Reporter runs in CI when REPORTER_API_URL is set (playwright.config.ts):

const useRocketiumReporter =
  process.env.CI && !process.env.DISABLE_REPORTER && !!reporterConfig.apiUrl?.trim();

For local runs with reporter, set CI=true or adjust that condition.

  UI->>API: GET /api/ui/projects/.../runs

  UI->>API: GET artifacts file

Quick Reference — Deployment Order

1. RDS PostgreSQL     → save DATABASE_URL
2. S3 artifacts       → save bucket name + IAM role
3. EC2 backend        → .env + pnpm start + Nginx
4. S3 + CloudFront UI → build dist/, upload, route /api
5. Dashboard          → create project, copy API key + project ID
6. GitHub secrets     → REPORTER_API_URL, REPORTER_API_KEY
7. CI workflow        → REPORTER_PROJECT_ID, shard vars
8. Verify             → /health + run workflow + check UI

13. Security Groups & Creation Order

13.1 Does S3 need a security group?

No. S3 is not inside your VPC and does not use security groups. Access is controlled by:

  • IAM role on EC2 (PutObject / GetObject)
  • Bucket policy (optional)
  • Block public access (keep ON)

CloudFront also has no VPC security group — it is a global edge service.

13.2 What to create first?

Yes — create security groups before (or while) creating ALB, EC2, and RDS. Recommended order:

  1. IAM role reporter-ec2-role — S3 access (no SG)
  2. S3 buckets — artifacts + UI static files (no SG)
  3. Security groups — create all 3 empty shells in the Default VPC:
    • sg-reporter-alb
    • sg-reporter-ec2
    • sg-reporter-rds
  4. RDS — attach sg-reporter-rds, Public access = No
  5. EC2 — attach sg-reporter-ec2 + IAM role
  6. Add SG rules — RDS ← EC2, EC2 ← ALB (see tables below)
  7. ALB — attach sg-reporter-alb, target group → EC2:3000
  8. CloudFront — origins: S3 UI + ALB
  9. Backend .env + GitHub secrets

13.7 AWS Console — create a security group

  1. EC2 → Security Groups → Create security group
  2. Name: e.g. sg-reporter-rds, VPC: Default VPC (same as RDS/EC2)
  3. Add inbound/outbound rules from tables above
  4. When referencing another SG, choose Source type = Security group and pick the peer SG by name
